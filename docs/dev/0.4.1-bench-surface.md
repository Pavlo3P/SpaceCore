# 0.4.1 Benchmark surface

Implementation checklist for the microbenchmark surface specified by
[ADR-023](adr/023_benchmark_framework.md). This is the **agreed scope** of *what
gets measured* and *what each operation is compared against*. Once an operation
is listed here it gets a probe whose `bare` callable is the **hand-optimal
pure-array-library** implementation (see ADR-023 "The pure-array-library
baseline").

## Scope decisions

Settled in review:

- **`linalg` module — out.** Iterative solvers are not in the micro surface.
- **`kernel` probe family — out as a *family*.** The optimized folds are not
  probed directly. They are still exercised: the `linop.block_diagonal` /
  `composed` / `stacked` / `sum_to_single` probes route through them under the
  `dispatch` / `dispatch+cache` regimes (below), so the kernel payoff is measured
  on the *real operator*, not a synthetic kernel probe.
- **`check_member` and similar — out.** Operations with no real array-library
  counterpart (the bare floor is a no-op) are not measured; they would report
  "pure abstraction overhead" with nothing to compare against.
- **Tree operations — out.** `space.tree.*` is composition of leaf methods: if
  the leaf methods are efficient, the tree op is efficient modulo `optree`
  traversal time. Measuring it adds no information beyond the leaf probes plus a
  fixed `optree` cost.
- **Block-structured operators — uniform AND ragged.** `block_diagonal`,
  `stacked`, and `sum_to_single` each get a **uniform-blocks** variant (where a
  batched-`matmul` fold exists) and a **ragged-blocks** variant (where it does
  not). The pair is the point: uniform shows the fold's win; ragged shows the
  abstraction adds ~nothing when no fast path applies.

Author rulings on the three open items (revise if disagreed):

- **`generated_{dense,diagonal}` — drop.** They duplicate `dense`/`diagonal`
  built via a generator; generator-construction is not a per-call hot path.
- **Elementwise-Jordan ops — keep, low priority.** `spectrum` /
  `spectral_decompose` / `star` / `from_spectrum` are mathematically thin on an
  elementwise algebra but exercise the real Jordan API path; kept, flagged P2.
- **Bare for ragged blocks** = idiomatic per-block Python loop (or
  `scipy.linalg.block_diag @ x` where assembling is the honest user choice), not
  a forced batched form.

## Configuration regimes

Every `linop` probe is timed under each regime so the ADR-016 dispatch and
ADR-022 caching wins are *measured*, not assumed. Regimes are driven by the real
API (`spacecore.kernels.dispatch_mode`, `set_memory_budget_fraction`):

| Regime | Dispatch | Materialized-form cache | How |
|---|---|---|---|
| `baseline` | off | inert | `dispatch_mode("off")` — today's default path |
| `dispatch` | on | cold each timed call | `dispatch_mode("on")` + per-call cache reset |
| `dispatch+cache` | on | warm (reused) | `dispatch_mode("on")`, operator reused across calls |
| `verify` | verify | warm | `dispatch_mode("verify")` — correctness, not headline speed |

**Cache reality (important).** SpaceCore has *no* cache on/off flag: the ADR-022
memo (`CachedStackParts`) builds lazily on first optimized use and lives on the
operator instance. Because `bench.harness.time_op` reuses one operator across all
repeats, the steady-state dispatch-on measurement *already* includes the cache.
To isolate the cache's contribution we therefore need two regimes:

- `dispatch+cache` is the natural reuse path (memo warm) — the full optimized
  result.
- `dispatch` forces a **cold cache each timed call**. ADR-022 documents that a
  pytree round-trip re-normalizes `parts` to a plain tuple and rebuilds an empty
  cache, so the runner resets the memo via that round-trip before each sample.
  This requires the probe to **expose its routed operator** to the runner (a
  small `ProbeCase` extension), so the cache can be reset without rebuilding the
  whole operator.

`space` and `functional` probes have no dispatch path and run `baseline` only.
The regime is part of every result row's coordinate and a group-by / pivot axis
in the dashboard.

## Implementation phases

1. **Regime infrastructure** — `bench/_regimes.py` (regime enum + per-regime
   context), `regime` field on `ProbeResult` (+ `_io` round-trip), runner sweep
   (`regimes=("baseline",)` default, non-baseline only for dispatch-eligible
   families), `regime_speedup` derived metric, smoke test. *Non-destructive.*
2. **Cache regime** — extend `ProbeCase` to expose the routed operator; runner
   resets the memo (pytree round-trip) before each timed sample for the
   `dispatch` regime; `dispatch+cache` reuses. Test the cold/warm distinction
   actually moves on a uniform block fold.
3. **Ragged block probes** — add ragged variants for `block_diagonal` /
   `stacked` / `sum_to_single` with per-block-loop bares.
4. **Bare-baseline audit** — verify every `bare` is hand-optimal per backend and
   matches the reference value (runner already records `bare` error); fix any
   strawman bares.
5. **HTML aggregation** — embed raw cells + client-side group-by/pivot
   (size / module / operation / backend / device / check_level / **regime**),
   geometric-mean speedup, worst-case min, correctness gate.
6. **Comparison / track** — `bench/_compare.py` join-then-diff at the group
   level (regime in the join key), seed-jitter noise band, environment guard;
   `track` for release-over-release trend.

Phases 1–4 are the measurement core; 5–6 are reporting. The out-of-scope
families have been removed from the suite to match this table: `linalg` probes,
the synthetic `kernel`-comparison probes (folds still measured via `linop` under
the dispatch regimes), `space.check_member`, the `space.tree.*` probes, and the
`linop.generated_{dense,diagonal}` probes are deleted; `functional.generated_linear.value`
and the elementwise-Jordan probes are kept. Final surface: `space` / `linop` /
`functional` only.
